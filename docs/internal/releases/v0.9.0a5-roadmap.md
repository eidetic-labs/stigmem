# v0.9.0a5 Release Roadmap

**Status:** historical
**Release owner:** maintainer
**Milestone:** `v0.9.0a5`
**Tag / release:** `v0.9.0a5`
**Last updated:** 2026-05-23

## Release Intent

`v0.9.0a5` validates the RTBF tombstone extraction in the alpha series. The
release goal is narrow: prove that tombstone routes, filters, issuance,
revocation, and federation handling are owned by the opt-in
`stigmem-plugin-tombstones` boundary, that default installs do not expose
tombstone behavior without plugin registration, and that the feature-owned spec,
security, evidence, compatibility, and release-note surfaces all describe the
same current state.

This release does not graduate tombstones into the default supported surface.
It also does not open beta, release-candidate, GA, stable-production readiness,
or standalone plugin artifact publication unless issue #298 is explicitly
moved into this milestone.

### Goal 2 - Validate the Tombstone Plugin Boundary

**Issue:** [#590](https://github.com/eidetic-labs/stigmem/issues/590)

**Objective:** prove RTBF tombstones are opt-in and unavailable in default
installs.

**Included changes:**
- Verify `experimental/tombstones/pyproject.toml` exposes the intended plugin
  entry point and package metadata.
- Verify `experimental/tombstones/src/stigmem_plugin_tombstones/manifest.py`
  declares only the required hooks, migrations, and capabilities.
- Verify tombstone config gates route and filter behavior off by default and
  honors explicit operator enablement.
- Verify default installs do not mount tombstone admin/federation routes or
  apply tombstone filters unless the plugin is registered.
- Verify plugin-loaded tests cover route/filter availability and expected
  hook behavior.

**Excluded / deferred:**
- Publishing `stigmem-plugin-tombstones` as a standalone registry artifact.
- Converting tombstones to a supported default surface.

**Evidence required:**
- Focused plugin scaffold, gating, and route/filter tests pass.
- Feature evidence references the exact implementation and test paths.

### Goal 3 - Validate Tombstone Issuance and Authority

**Issue:** [#591](https://github.com/eidetic-labs/stigmem/issues/591)

**Objective:** prove tombstones cannot be issued, accepted, or propagated
without the required authority and verification evidence.

**Included changes:**
- Validate tombstone IDs, signed record shape, scope pattern handling, issuing
  admin identity, reason code, and audit metadata against
  `Spec-X2-RTBF-Tombstones`.
- Validate tombstone issuance by a non-admin key returns 403.
- Validate forged tombstone signatures are rejected.
- Validate federation inbound tombstones without tombstone authority are
  rejected and audited.
- Add or tighten tests where current coverage only proves scaffold presence.

**Excluded / deferred:**
- Production legal workflow design for right-to-be-forgotten requests.
- ADR-008 graduation or stable compliance claims.

**Evidence required:**
- Focused security and federation tests pass.
- `features/tombstones/evidence.md` lists the passing test commands.

### Goal 4 - Validate Filtering, Revocation, and Legal-Hold Behavior

**Issue:** [#593](https://github.com/eidetic-labs/stigmem/issues/593)

**Objective:** prevent tombstones from leaking suppressed state or becoming a
recovery/destructive-action path.

**Included changes:**
- Validate facts, recall results, and memory cards suppress matching tombstoned
  entities and graph references when the plugin is enabled.
- Validate tombstone revocation restores only the intended read surface.
- Validate non-admin `as_of` reads cannot cross a legal-hold tombstone
  boundary.
- Validate admin/legal-hold behavior stays aligned with the a4 time-travel
  boundary without making tombstones an a4-owned feature.
- Validate response counts, headers, and error bodies do not reveal legal-hold
  existence to non-admin callers.

**Excluded / deferred:**
- Reworking the core recall pipeline beyond the tombstone hook/filter path.
- Making legal-hold historical access a stable compatibility commitment.

**Evidence required:**
- Tombstone behavior and legal-hold tests pass.
- `features/tombstones/security.md`, `SECURITY.md`, and release notes agree on
  residual risk and operator posture.

## Security and Risk Posture

`v0.9.0a5` remains an alpha release. RTBF tombstones are experimental and
opt-in. The main security invariant is fail-closed suppression: default
installs must not mount tombstone routes or apply tombstone filters, non-admin
callers must not learn legal-hold state, forged or unauthorized tombstones must
be rejected, and operators must explicitly enable the feature.

Security publication policy remains unchanged: Critical/High disclosures use
GHSA where applicable; Medium/Low findings remain in `SECURITY.md` unless a
documented risk-profile, reporter-coordination, or downstream-compliance
carve-out applies.
