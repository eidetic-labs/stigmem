# v0.9.0a4 Release Roadmap

**Status:** active
**Release owner:** maintainer
**Milestone:** `v0.9.0a4`
**Tag / release:** not tagged
**Last updated:** 2026-05-22

## Release Intent

`v0.9.0a4` validates the time-travel query extraction in the alpha series. The
release goal is narrow: prove that `as_of` historical reads are owned by the
opt-in `stigmem-plugin-time-travel` boundary, that default installs fail closed
without plugin registration, and that the feature-owned spec, security,
evidence, compatibility, and release-note surfaces all describe the same
current state.

This release does not graduate time-travel into the default supported surface.
It also does not open beta, release-candidate, GA, or stable-production
readiness gates.

### Goal 2 — Validate the Plugin Boundary

**Issue:** [#571](https://github.com/eidetic-labs/stigmem/issues/571)

**Objective:** prove time-travel is opt-in and unavailable in default installs.

**Included changes:**
- Verify `experimental/time-travel/pyproject.toml` exposes the intended plugin
  entry point and package metadata.
- Verify `experimental/time-travel/src/stigmem_plugin_time_travel/manifest.py`
  declares only the required hooks and capabilities.
- Verify `TimeTravelConfig` gates fact-query and recall `as_of` behavior off
  by default and honors explicit operator enablement.
- Verify default installs return `time_travel_plugin_not_loaded` for `as_of`
  requests.
- Verify plugin-loaded tests cover both fact query and recall historical reads.

**Excluded / deferred:**
- Publishing `stigmem-plugin-time-travel` as a standalone registry artifact.
- Converting the feature to a supported default surface.

**Evidence required:**
- Focused plugin scaffold and validation tests pass.
- Feature evidence references the exact implementation and test paths.

### Goal 3 — Validate Historical Query Semantics

**Issue:** [#572](https://github.com/eidetic-labs/stigmem/issues/572)

**Objective:** prove the `as_of` behavior matches `Spec-X3-Time-Travel-Queries`.

**Included changes:**
- Validate timestamp parsing and error codes:
  `as_of_invalid_timestamp`, `as_of_future`, and
  `as_of_before_retention_floor`.
- Validate fact visibility rules: created before `T`, not expired at `T`, not
  retracted at or before `T`, and not covered by a disallowed tombstone.
- Validate recall results are scored relative to the requested historical time.
- Validate CID/source-trust interaction does not bypass historical visibility
  or tamper-detection behavior.
- Add or tighten tests where existing coverage is only indirect.

**Excluded / deferred:**
- Changing the core recall scoring model beyond the time-travel path.
- Making retention policy a stable compatibility commitment.

**Evidence required:**
- `node/tests/time_travel/test_phase13_time_travel_cid.py` or successor tests
  cover the release claims.
- `features/time-travel/evidence.md` lists the passing test commands.

### Goal 4 — Validate Legal-Hold, Tombstone, and Security Posture

**Issue:** [#573](https://github.com/eidetic-labs/stigmem/issues/573)

**Objective:** prevent time-travel from becoming a historical-data disclosure
path.

**Included changes:**
- Validate non-legal-hold tombstones suppress matching facts retroactively.
- Validate legal-hold facts are visible only to the authorized admin path.
- Validate agent/non-admin callers do not receive a response that reveals
  legal-hold existence.
- Align `features/time-travel/security.md`, `SECURITY.md` projections, and the
  threat-model references for R-17 and R-18.
- Keep the a5 RTBF tombstone launch distinct from the a4 time-travel boundary.

**Excluded / deferred:**
- Closing every tombstone feature gap in a4.
- Opening a stable compliance claim for legal-hold historical access.

**Evidence required:**
- Focused security tests pass.
- Security docs state the residual risk and operator posture without
  duplicating the full feature spec.

## Security and Risk Posture

`v0.9.0a4` remains an alpha release. Time-travel is experimental and opt-in.
The main security invariant is fail-closed historical access: default installs
must reject `as_of`, non-admin callers must not learn legal-hold state, and
operators must explicitly enable historical reads.

Security publication policy remains unchanged: Critical/High disclosures use
GHSA where applicable; Medium/Low findings remain in `SECURITY.md` unless a
documented risk-profile, reporter-coordination, or downstream-compliance
carve-out applies.
