**Status:** active

**Tag / release:** not tagged
**Last updated:** 2026-05-23

## Release Intent

`v0.9.0a8` validates the multi-tenant isolation extraction in the alpha
series. The feature resolves API-key tenant identity into a tenant-scoped
request context only when operators explicitly register and enable
`stigmem-plugin-multi-tenant`. Default installs continue to collapse callers
into the single `default` tenant.

This release is a validation and artifact-refresh horizon. It completes the
planned alpha extraction train, but it does not graduate multi-tenant scoping
into the supported default surface, open beta/RC/GA, or publish standalone
plugin artifacts unless issue #298 is explicitly moved into this milestone.

## Security and Risk Posture

`v0.9.0a8` remains an alpha release. Multi-tenant scoping is experimental and
opt-in. The main security invariant is boundary honesty: default installs must
remain single-tenant, plugin-loaded deployments must preserve tenant context
across sensitive surfaces, and docs must not present the alpha plugin as stable
shared-node readiness.

Multi-tenant scoping contributes to R-01, R-02, and R-21 because shared-node
deployments amplify peer-authentication, quota, and feedback-loop risks. This
release does not close those risks. Promotion requires tenant-aware
conformance vectors, an explicit Spec-X decision if needed, per-tenant quota
evidence, external operator soak evidence, and an explicit risk-ownership
decision for tenant-isolation failures.

Security publication policy remains unchanged: Critical/High disclosures use
GHSA where applicable; Medium/Low findings remain in `SECURITY.md` unless a
documented risk-profile, reporter-coordination, or downstream-compliance
carve-out applies.
