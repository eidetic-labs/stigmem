# v0.9.0a3 Release Roadmap

**Status:** active
**Release owner:** maintainer
**Milestone:** `v0.9.0a3`
**Tag / release:** not tagged
**Last updated:** 2026-05-22

## Release Intent

`v0.9.0a3` is the active alpha horizon after the `v0.9.0a2` artifact
refresh. Its defined scope is CID core/spec validation and the next alpha
artifact refresh.

The release must keep the public posture aligned with reality: Stigmem remains
alpha, beta/RC/GA milestones stay unopened, and plugin artifact publication
remains on the plugin launch train unless the maintainer explicitly moves it
into this milestone.

## Detailed Scope

### CID Core/Spec Validation

**Objective:** close the active alpha validation loop for content-addressed IDs
as a core feature.

**Included changes:**
- Confirm the CID feature record, spec, status, changelog, evidence, and
  security files describe the same release status.
- Confirm `docs/compatibility-matrix.yaml` projects the CID feature into
  `v0.9.0a3`.
- Validate that the modular spec projection for content-addressed IDs matches
  the feature-owned source material.
- Keep CID documented as core per ADR-017; no plugin package is planned for
  this feature.

**Excluded / deferred:**
- Creating or publishing a CID plugin artifact.
- Moving CID code or docs out of the core feature boundary.
- Treating CID validation as a beta, RC, or GA readiness claim.

**Evidence required:**
- Feature documentation checks pass for `features/content-addressed-ids/`.
- Compatibility matrix and spec projection checks pass.
- Release notes name the CID validation work that actually ships.

surfaces Stigmem currently supports.

**Included changes:**
- Prepare versioned core Python package artifacts for `v0.9.0a3`.
- Prepare versioned npm/TypeScript SDK artifacts for `0.9.0-alpha.3`.
- Prepare the OpenClaw adapter artifact aligned to the a3 release.

**Excluded / deferred:**

- Beta, RC, or GA version strings.
- Claims that alpha artifacts are production-stable.

**Evidence required:**
- Package build and release validation passes for changed artifact surfaces.
- Docs build passes.
- Release notes reconcile with `CHANGELOG.md` and this roadmap.
- Tag and GitHub release evidence is captured at release closeout.

### Security and Advisory Documentation Discipline

**Objective:** keep the security publication model aligned with established
policy while preparing a3.

**Included changes:**
- Critical/High disclosures remain GHSA-backed when applicable.
- Medium/Low dispositions remain in `SECURITY.md` unless the risk profile,
  reporter coordination, or downstream compliance need triggers a policy
  carve-out.
- Dated evidence registries remain proof ledgers, not public advisory indexes.
- Internal-Comms remains staging only for embargoed material.

**Excluded / deferred:**
- Future-publication language for findings already documented in `SECURITY.md`.
- Duplicating the same advisory table across PR bodies, changelog, evidence
  ledgers, and posture docs.

**Evidence required:**
- Security documentation registry check passes.
- Evidence registry validators pass for changed mitigated risks.
- Release notes and `SECURITY.md` do not contradict each other.

**References:**
- `SECURITY.md`
- `docs/internal/security-evidence-registry-2026-05-17.md`

## Security and Risk Posture

`v0.9.0a3` remains an alpha release. It must not claim hardened-core,
production federation, beta, RC, or GA readiness. Security work in this horizon
must preserve the publication policy: Critical/High through GHSA where
applicable; Medium/Low in `SECURITY.md` unless a documented policy carve-out
applies.
