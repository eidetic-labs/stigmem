# v0.9.0a3 Release Roadmap

**Status:** active
**Release owner:** maintainer
**Milestone:** `v0.9.0a3`
**Tag / release:** not tagged
**Last updated:** 2026-05-21

## Release Intent

`v0.9.0a3` is the active alpha horizon after the `v0.9.0a2` artifact refresh.
Its purpose is to restore release-planning discipline, preserve the strategic
roadmap detail that was compressed during docs cleanup, and carry the next
alpha artifact/evidence work without opening beta, RC, or GA prematurely.

This release should make the alpha scope explicit enough that "implement X"
always decomposes into owned workstreams, evidence gates, and release-note
candidates.

## Detailed Scope

### Release Documentation Architecture

**Objective:** create a durable planning structure so future releases do not
depend on one long, mixed-purpose roadmap.

**Included changes:**
- Define the standard format for `ROADMAP.md`.
- Define the standard format for internal release roadmaps.
- Add historical release roadmaps for `v0.9.0a1` and `v0.9.0a2`.
- Add this active `v0.9.0a3` release roadmap.
- Update documentation ownership so roadmap, release scope, readiness, and
  release cadence have distinct canonical homes.

**Excluded / deferred:**
- ADR edits.
- Rewriting git history or moving historical tags.

**Evidence required:**
- Internal docs link consistently to the new standards.
- No ADR files changed.
- Docs build remains green.

**References:**
- `docs/internal/roadmap-standards.md`
- `docs/internal/documentation-ownership.md`
- `docs/internal/releases/`

### Roadmap Restoration

**Objective:** restore the strategic detail that existed in the older roadmap
without reintroducing stale active milestone claims.

**Included changes:**
- Preserve version-line goals for alpha, future hardened-core, future RC/GA,
  and post-GA expansion.
- Preserve the old roadmap's detailed workstreams: capability redesign,
  certification, storage immutability, security colocation, federation
  hardening, Argon2id, operator docs, quality gates, contributor/demo
  readiness, external operator soak, supply-chain evidence, external
  organizations, compatibility, plugin stewardship, source/test hygiene, and
  evidence maintenance.
- Link detailed workstreams to internal release or future-horizon docs instead
  of leaving them as vague one-line roadmap items.

**Excluded / deferred:**
- Marking beta, RC, or GA as active.
- Creating beta, RC, or GA milestones before their entry gates are satisfied.

**Evidence required:**
- Diff against prior roadmap revisions shows workstreams preserved or moved.
- `ROADMAP.md` explains where detailed release contracts live.

**References:**
- Historical `ROADMAP.md` revisions in git history.
- `ROADMAP.md`

**Included changes:**

**Evidence required:**

**References:**

- `docs/internal/release-cadence.md`

### Security and Advisory Documentation Discipline

**Objective:** keep the security publication model aligned with policy.

**Included changes:**
- Critical/High disclosures remain GHSA-backed when applicable.
- Medium/Low dispositions remain in `SECURITY.md` unless risk profile changes.
- Dated evidence registries remain proof ledgers, not public advisory indexes.
- Internal-Comms remains staging only for embargoed material.

**Excluded / deferred:**
- Future-publication framing for findings already documented in `SECURITY.md`.
- Duplicating the same advisory table across PR bodies, changelog, evidence
  ledgers, and security posture docs.

**Evidence required:**
- Security documentation registry check passes.
- Evidence registry validators pass for changed mitigated risks.

**References:**
- `SECURITY.md`
- `docs/internal/security-evidence-registry-2026-05-17.md`
- `docs/internal/documentation-ownership.md`

## Security and Risk Posture

`v0.9.0a3` remains an alpha release. It should not claim hardened-core,
production federation, beta, RC, or GA readiness. Security work in this horizon
must preserve the publication policy: Critical/High through GHSA where
applicable; Medium/Low in `SECURITY.md` unless risk profile changes.
