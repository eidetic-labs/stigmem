# Stigmem v2.0 Release Notes

**Published:** 2026-05-05  
**Spec:** [stigmem-spec-v2.0.md](https://docs.stigmem.dev/next/spec/)  
**Audience:** Node operators, SDK consumers, and protocol implementers.

---

## Summary

Stigmem v2.0 promotes seven specification sections (§19–§25) from draft to normative, completing the protocol's federation security, semantic recall, data protection, and content-integrity layers.

This release is the result of eight development phases (8–14) spanning storage-backend abstraction, federation trust, graph-based recall, lazy instruction discovery, security hardening, RTBF compliance, time-travel queries, and content-addressed fact IDs.

---

## What shipped

### Phase 8 — Storage adapter trait + libSQL/Turso backend

- `StorageBackend` trait abstracting connection lifecycle, transactions, and migrations.
- `LibSQLBackend` for libSQL/Turso embedded-replica deployments.
- Backend-parameterized test runner (`pytest --backend=libsql`).

### Phase 9 — Graph memory & recall (§20)

- Materialized `entity_edges` graph index for O(edges) k-hop traversal.
- Vector embeddings via `sqlite-vec` with `nomic-embed-text-v1.5` (768-dim, Apache-2.0).
- Hybrid recall pipeline: lexical BM25 + dense ANN + graph BFS with MMR packing.
- Memory cards — per-entity synthesized summaries (4000-token cap).
- Subscription primitive with webhook and agent-wake delivery.
- Causal/derivation links (`derived_from` DAG on fact records).

### Phase 10 ��� Lazy instruction discovery (§21)

- Boot stub (≤500 tokens) + instruction manifest (≤1000 tokens).
- `recall_instruction` tool contract backed by stigmem recall on `instruction:` scope.
- Task-type preloads with `guarantee_load` flag (max 5 per agent).
- Discovery audit metrics: Recall@k, Hit@k, miss-rate.
- 5-stage file → stigmem migration path with `stigmem instruction migrate` CLI.

### Phase 11 — Federation trust (§19)

- Org manifest signing with Ed25519 + RFC 8785 JCS canonicalization.
- Transparency log integration (Rekor-compatible; inclusion proofs, signed checkpoints).
- Capability tokens: `read`, `write`, `admin`, `federate`, `subscribe`, `tombstone:read` verbs; 90-day expiry ceiling.
- Source-trust scoring (0.0–1.0) with quarantine garden for low-trust facts.
- Recall-time content sanitizer with configurable deny patterns.

### Phase 12 — Security hardening (§22)

- Mutual TLS required for all federation transport (TLS 1.3 floor).
- Ed25519 key rotation with 90-day dual-trust period and transparency log entries.
- Structured audit log: 13 mandatory event types, write-ahead ordering, 90-day retention floor.
- Per-principal token-bucket quotas (7 dimensions; HTTP 429 backpressure).
- Replay protection: ±5 minute timestamp window with persistent nonce cache.
- Container baseline: distroless image, non-root UID 1000, read-only filesystem, seccomp.

### Phase 13 — RTBF, time-travel, and content-addressed IDs (§23–§25)

- **RTBF tombstones (§23):** Signed tombstone records; recall-time and graph-reference suppression; federation propagation with inbound verification; tombstone revocation for legal holds.
- **Time-travel queries (§24):** `as_of` parameter on `/v1/recall` and `/v1/facts`; fact visibility at historical timestamps; append-only `fact_retractions` log; legal-hold annotation for admin-only historical access.
- **Content-addressed IDs (§25):** `CID = "sha256:" + hex(SHA-256(RFC8785(canonical_body)))` over 6 canonical fields; `fact_cid_aliases` table for dual UUID/CID addressing; 12-month migration window; federation tamper detection; `stigmem backfill-cids` CLI.

### Phase 14 — Spec v2.0 publication + conformance

- All §19–§25 sections promoted from draft to normative.
- OpenAPI spec version bumped to 2.0.0.
- v2.0 conformance vectors published for §19–§25.
- Documentation site information architecture restructured.
- Migration guide and release notes (this document).

## Conformance

The v2.0 conformance suite (`data/conformance/v2.0/`) covers all normative sections. CI enforces zero-skip conformance on every push to `main`.

Conformance vectors for §19–§25 test:
- Federation handshake and capability-token verification.
- Recall pipeline ranking and graph traversal bounds.
- Tombstone issuance, recall-time filter, and federation propagation.
- Time-travel fact visibility and retraction semantics.
- CID computation, dual-addressing, and tamper detection.

## Known caveats

1. **Embedding model lock-in.** Changing `STIGMEM_EMBED_DIMENSIONS` after indexing requires a full re-index (`stigmem reindex-embeddings`). Plan your embedding model choice before production deployment.

2. **Mixed-version federation.** v1.x and v2.0 nodes cannot federate. All nodes in a federation cluster must upgrade together.

3. **CID backfill duration.** Large fact stores (>1M facts) may take hours to backfill CIDs. The backfill runs concurrently with live writes and does not block normal operation.

4. **Tombstone propagation latency.** Federation peers may take up to 5 minutes to receive a new tombstone via polling. The subscription channel (`tombstone_new` event) reduces this to ~60 seconds.

5. **Transparency log dependency.** In `trust_mode: strict`, federation connections fail if the transparency log is unreachable. Operators should monitor log availability.
